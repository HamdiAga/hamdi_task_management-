كيف تنقل مشروعك من بيئة التطوير المحلية إلى الإنتاج دون أن ينفجر في وجهك

10 يونيو 2026 حمدي فواز أبو حاتم Developer Full-Stack

كل مطوّر لديه قصة نشر كارثية. السيرفر لا يستجيب، قاعدة البيانات لا تتصل، الصور مختفية، والأسوأ: العميل يتصل غاضباً. النشر إلى الإنتاج (Deployment) هو اللحظة التي تتحول فيها شهور من العمل إلى منتج حي. سأشاركك طريقي الآمن لنقل مشاريع Laravel وNode.js إلى الإنتاج باستخدام Laravel Forge وVPS، مع تفادي الأخطاء الشائعة.

1. القاعدة الذهبية: لا تنشر مباشرة من جهازك!

أكبر خطأ: ترفع الملفات عبر FTP من مجلد المشروع المحلي. هذا يؤدي إلى:

  • نسيان ملفات (مثل .env أو مجلدات مخفية).
  • رفع ملفات غير ضرورية (node_modules، .git، ملفات اختبار).
  • عدم وجود أثر للنسخة المنشورة (لا تعرف أي نسخة تعمل على السيرفر).

الحل: استخدم Git. كل نشر يجب أن يكون من مستودع Git (GitHub, GitLab).

2. خطوات النشر الآمن لمشروع Laravel

  1. جهّز السيرفر: استخدم Laravel Forge (أو aaPanel المجاني) لتجهيز السيرفر تلقائياً: Nginx, PHP 8.2, MySQL 8, Redis, Composer, Node.js.
  2. اربط المستودع: في Forge، اربط موقعك بمستودع GitHub. كل Push يبني المشروع تلقائياً.
  3. إعدادات .env: لا ترفع .env الحقيقي إلى Git. في السيرفر، أنشئ .env بالإعدادات الفعلية (اتصال قاعدة البيانات، مفاتيح البوابات).
  4. تشغيل الأوامر الأساسية:
git pull origin main
composer install --no-dev --optimize-autoloader
php artisan migrate --force
php artisan config:cache
php artisan route:cache
php artisan view:cache
npm install && npm run build
  1. تأكد من الصلاحيات: مجلد storage/ و bootstrap/cache/ يجب أن يكونا قابلين للكتابة.
  2. أعد تشغيل Queue Workers: بعد كل نشر، أعد تشغيل العاملين: php artisan queue:restart.

3. خطوات النشر الآمن لمشروع Node.js

  1. PM2: أداة لإدارة عملية Node.js. بعد رفع المشروع، شغّله بـ: pm2 start app.js --name "my-api"
  2. Nginx Reverse Proxy: وجّه النطاق إلى تطبيق Node.js الذي يعمل على منفذ 3000.
  3. متغيرات البيئة: استخدم ملف .env وحمّله في بداية التطبيق: require('dotenv').config();

4. قائمة تدقيق ما قبل النشر (Pre-Deployment Checklist)

  • هل APP_DEBUG=false؟ لا تترك debug في الإنتاج، سيكشف معلومات حساسة.
  • هل APP_ENV=production؟ هذا يغير سلوك Laravel (تكييش، تقارير أخطاء).
  • هل قواعد البيانات محدثة؟ شغّل migrations وتأكد من عدم وجود pending migrations.
  • هل الـ Storage مرتبط؟ إذا كنت تستخدم S3 أو Spaces، تأكد من المفاتيح.
  • هل اختبرت على Staging؟ استخدم بيئة staging (نسخة مطابقة للإنتاج) واختبر كل شيء قبل النشر النهائي.

خطأ كلفني ساعات: في أحد المشاريع، نسيت تشغيل php artisan storage:link بعد النشر، فاختفت كل صور المنتجات من الموقع. العميل اتصل مذعوراً. الحل كان سطر أوامر واحداً، لكن الدرس بقي: اختبر كل شيء بعد النشر مباشرة.

5. ماذا تفعل إذا انفجر الموقع بعد النشر؟

  1. لا داعي للذعر. أول شيء: تحقق من logs (storage/logs/laravel.log أو pm2 logs).
  2. تراجع فوراً: استخدم Git للعودة إلى آخر نسخة عاملة: git revert HEAD~1 ثم نشر.
  3. حافظ على هدوئك مع العميل: قل له "نعمل على تحسين طارئ، الموقع سيعود خلال دقائق." ولا تختفي.

خلاصة

النشر إلى الإنتاج ليس مخيفاً إذا اتبعت عملية واضحة. استخدم Git، جهّز السيرفر بأداة مثل Forge أو aaPanel، استخدم بيئة staging، واحتفظ بقائمة تدقيق. والأهم: بعد كل نشر، افتح الموقع بنفسك وتأكد من أن كل شيء يعمل.
