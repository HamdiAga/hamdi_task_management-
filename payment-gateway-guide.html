دليل المبرمج للتعامل مع بوابات الدفع: من PayPal إلى Stripe إلى الدفع المحلي

10 يونيو 2026 حمدي فواز أبو حاتم Developer Full-Stack

كل متجر إلكتروني يحتاج إلى بوابة دفع. لكن اختيار البوابة المناسبة وطريقة ربطها بشكل صحيح هو ما يفرق بين مطوّر مبتدئ ومطوّر محترف. في هذا الدليل العملي، سأشرح لك كيفية التعامل مع PayPal وStripe وبوابات الدفع المحلية في الخليج، مع التركيز على webhooks والأخطاء الشائعة وتجربتي الشخصية.

1. فهم تدفق الدفع الإلكتروني

أي بوابة دفع تعمل بنفس التدفق الأساسي:

  1. المستخدم يضغط "ادفع الآن".
  2. الـ Frontend يرسل مبلغ العملية ومعرّف الطلب إلى الـ Backend.
  3. الـ Backend ينشئ جلسة دفع على بوابة الدفع ويستلم redirect_url أو token.
  4. يُوجّه المستخدم إلى صفحة الدفع الآمنة على بوابة الدفع.
  5. بعد إتمام الدفع، ترسل البوابة Webhook إلى عنوان خاص في الـ Backend.
  6. الـ Backend يتحقق من صحة الـ Webhook ويُحدّث حالة الطلب إلى "مدفوع".

2. PayPal – التكامل الكلاسيكي

PayPal يوفر SDKs جاهزة، لكني أفضل التعامل مع REST API مباشرة للتحكم الكامل. خطوات الربط:

  • احصل على Client ID و Secret من PayPal Developer Dashboard.
  • أنشئ Order عبر API:
POST /v2/checkout/orders
{
  "intent": "CAPTURE",
  "purchase_units": [{
    "amount": { "currency_code": "SAR", "value": "100.00" }
  }]
}
  • استلم redirect_url ووجّه المستخدم.
  • بعد العودة، نفّذ capture عبر API لتحصيل المبلغ فعلياً.

نصيحة: استخدم PayPal Sandbox بكثرة أثناء التطوير. ولا تنسَ تفعيل IPN (Instant Payment Notification) كطبقة احتياطية بجانب Webhooks.

3. Stripe – المفضل للمطورين

Stripe يتميز بوثائقه الممتازة وSDK القوي. التكامل عبر Stripe Checkout أو Payment Intents:

// Node.js مثال
const stripe = require('stripe')('sk_test_...');
const session = await stripe.checkout.sessions.create({
  payment_method_types: ['card'],
  line_items: [{
    price_data: {
      currency: 'sar',
      product_data: { name: 'منتج تجريبي' },
      unit_amount: 10000, // بالهللات
    },
    quantity: 1,
  }],
  mode: 'payment',
  success_url: 'https://mysite.com/success',
  cancel_url: 'https://mysite.com/cancel',
});

Stripe يعطيك جلسة، توجّه المستخدم إلى رابط الجلسة، وبعد الدفع تستلم Webhook لتأكيد العملية. أهم شيء: تحقق من توقيع Webhook باستخدام مفتاح signing secret.

4. الدفع المحلي في الخليج

في السعودية، بوابات مثل مدى (عبر HyperPay أو Tap) وتمارا وSTC Pay هي الأساس. تحدثنا عن تفاصيلها في مقال "ربط أنظمة المدفوعات الإلكترونية في السعودية والخليج". لكن أضيف هنا: عند التعامل مع بوابات محلية، انتبه لـ:

  • العملة: دائماً SAR، وتأكد من دعم الهللات.
  • اللغة: أرسل locale=ar لواجهة عربية.
  • رقم الجوال: بعض البوابات تتطلب إرسال رقم الجوال بصيغة 9665xxxxxxxx لإرسال رموز تأكيد.

5. إدارة Webhooks والتحقق من الأمان

الـ Webhook هو أساس تكامل الدفع. بدونه، لن تعرف أن المستخدم دفع. أفضل الممارسات:

  • تحقق من التوقيع (HMAC signature) دائماً.
  • سجّل كل Webhook في جدول payment_webhooks قبل معالجته.
  • إذا فشلت المعالجة، حاول مرة أخرى (retry) لكن مع منع التكرار (idempotency).
  • استخدم queues لمعالجة Webhooks الثقيلة (مثل إرسال إشعارات).
// مثال Laravel للتحقق من توقيع Stripe Webhook
$payload = $request->getContent();
$sig_header = $request->header('Stripe-Signature');
try {
    $event = \Stripe\Webhook::constructEvent(
        $payload, $sig_header, config('services.stripe.webhook_secret')
    );
} catch (\Exception $e) {
    return response()->json(['error' => 'Invalid signature'], 400);
}

6. أخطاء شائعة يجب تجنبها

  • عدم اختبار Sandbox كافياً: اختبر كل سيناريو: دفع ناجح، فشل، إلغاء، استرداد.
  • تخزين بيانات البطاقة: لا تفعلها أبداً. استخدم tokens.
  • عدم التعامل مع فرق التوقيت: أحياناً Webhook يصل قبل أن يكتمل تحديث الصفحة، فتظهر حالة "قيد الانتظار" للمستخدم. استخدم AJAX polling أو WebSocket.
  • نسيان اختبار الاسترداد (Refund): بعض البوابات المحلية تتطلب خطوات إضافية للاسترداد الجزئي.

خلاصة

بوابات الدفع هي شريان المتجر. تعلم كيف تتعامل معها باحترافية، وركز على الأمان (Webhooks signature) وتجربة المستخدم بعد الدفع. ابدأ ببوابة واحدة، أتقنها، ثم أضف غيرها حسب احتياج السوق.
