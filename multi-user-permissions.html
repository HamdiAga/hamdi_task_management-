كيف تنشئ نظام صلاحيات متعدد المستخدمين (Admin, Editor, Customer) بأمان من الصفر

9 يونيو 2026 حمدي فواز أبو حاتم Developer Full-Stack

كل متجر إلكتروني أو تطبيق يحتاج إلى نظام صلاحيات. مالك المتجر لا يمكنه إعطاء موظف الطلبات صلاحية تغيير الأسعار، والمندوب لا يجب أن يرى تقارير الأرباح. في هذا المقال، سأشرح لك كيف تبني نظام صلاحيات متدرج (RBAC – Role-Based Access Control) من الصفر، مع أمثلة من Laravel وNode.js، من واقع تجربتي مع متاجر سلة وزِد.

1. لماذا RBAC وليس مجرد عمود is_admin؟

عمود is_admin (true/false) يعطيك حالتين فقط. لكن ماذا لو احتجت 4 أدوار أو 10 أدوار بصلاحيات متداخلة؟ نظام RBAC يفصل بين الأدوار (Roles) والصلاحيات (Permissions)، مما يمنحك مرونة لا نهائية:

  • Admin: كل الصلاحيات.
  • Editor: إدارة المنتجات والمحتوى، بدون الوصول للتقارير المالية.
  • Order Manager: رؤية الطلبات وتحديث حالتها فقط.
  • Customer: رؤية طلباته الخاصة فقط.

2. هيكل قاعدة البيانات

التصميم الكلاسيكي لـ RBAC يستخدم 5 جداول:

users: id, name, email, ...
roles: id, name (admin, editor, ...)
permissions: id, name (edit-products, view-orders, ...)
role_permission: role_id, permission_id (الربط بين الأدوار والصلاحيات)
user_role: user_id, role_id (الربط بين المستخدمين والأدوار)

هذا الهيكل يسمح لك بإعطاء أكثر من دور للمستخدم الواحد، وأكثر من صلاحية للدور الواحد. في Laravel، استخدم حزمة spatie/laravel-permission لتوفير الوقت. لكني سأريك كيف تبنيها يدوياً.

3. البناء في Laravel – Middleware مخصص

// middleware/CheckPermission.php
public function handle($request, $next, $permission) {
    if (!auth()->user()->hasPermission($permission)) {
        abort(403, 'غير مصرح لك بهذا الإجراء');
    }
    return $next($request);
}

// في User model:
public function hasPermission($permission) {
    return $this->roles()->whereHas('permissions', function($q) use ($permission) {
        $q->where('name', $permission);
    })->exists() || $this->hasRole('admin');
}

// في routes/api.php:
Route::middleware('auth:sanctum')->group(function() {
    Route::get('/orders', [OrderController::class, 'index'])
         ->middleware('permission:view-orders');
    Route::post('/products', [ProductController::class, 'store'])
         ->middleware('permission:create-products');
});

4. في Node.js – Middleware مع JWT

// middleware/permission.js
function checkPermission(permission) {
    return (req, res, next) => {
        const user = req.user;
        if (user.roles.some(role => role === 'admin')) return next();
        if (user.permissions.some(perm => perm === permission)) return next();
        return res.status(403).json({ error: 'غير مصرح' });
    };
}

// usage
app.get('/orders', authenticate, checkPermission('view-orders'), orderController.list);

5. حماية البيانات – صلاحيات على مستوى الصف

الصلاحية لا تقتصر على "هل يرى صفحة الطلبات؟". أحياناً تريد أن يرى المندوب طلباته هو فقط:

// Laravel Policy
class OrderPolicy {
    public function view($user, $order) {
        if ($user->hasRole('admin')) return true;
        if ($user->hasRole('delivery') && $order->delivery_id === $user->id) return true;
        if ($user->hasRole('customer') && $order->user_id === $user->id) return true;
        return false;
    }
}

6. مثال من السوق – لوحة تحكم متجر سلة

في أحد مشاريع تطوير لوحة تحكم لمتجر سلة، كان صاحب المتجر يريد:

  • موظف طلبات يرى الطلبات ويغير حالتها فقط.
  • مصمم منتجات يضيف منتجات وصور بدون لمس الأسعار.
  • محاسب يرى التقارير المالية فقط.

بنينا نظام RBAC بالكامل. في الأسبوع الأول، اكتشفنا أن المصمم كان يغير الأسعار "بالخطأ" قبل النظام الجديد. بعد تطبيق الصلاحيات، اختفت هذه المشكلة تماماً، وأصبح صاحب المتجر ينام مرتاحاً.

نصائح أمنية

  • لا تثق بالـ Frontend أبداً. الصلاحيات يجب أن تتحقق دائماً في الـ Backend.
  • استخدم Gates و Policies في Laravel، فهي أنظف من فحص الصلاحيات يدوياً في كل Controller.
  • اختبر الصلاحيات بكتابة Feature Tests تحاول فيها الوصول إلى موارد بدون صلاحية وتتأكد من ظهور 403.

خلاصة

نظام الصلاحيات ليس كمالياً، بل هو أساس الأمان في أي تطبيق متعدد المستخدمين. استثمر وقتاً في تصميمه جيداً من البداية، لأنه إعادة هيكلته لاحقاً أصعب بكثير.
